## 1. Threat Model

### 1.1 Assets to protect

| Asset | Priority |
|---|---|
| User conversation data (PII possible) | High |
| Enterprise documents accessed by agents | High |
| vLLM backend (GPU-expensive) | High — abuse cost |
| User API keys | High |
| Admin API keys | Critical |
| Agent tools' upstream credentials | High |
| Audit logs | High (integrity) |

### 1.2 Threat actors

| Actor | Capabilities | Primary threats |
|---|---|---|
| External unauthenticated | Public internet access | DoS, credential stuffing, endpoint enumeration |
| Authenticated user (enterprise employee) | Valid API key, normal user scope | Over-use (cost abuse), prompt injection attempting privilege escalation, PII exfiltration attempts |
| Malicious agent author (insider with commit access) | Can introduce backdoored agent | Covered by code review; out of scope for this doc |
| Compromised external service (an MCP server's upstream) | Returns malicious tool results | Prompt injection via tool results |
| Ops/admin account compromise | Admin key stolen | Data exfil, service disruption |

### 1.3 Out of scope for v1.2

- **Network-level attacks** (assume on-prem behind corporate firewall + TLS termination upstream)
- **Physical access** (assume hardened datacenter)
- **Supply-chain attacks on Python packages** (mitigated partially via pip-audit + Dependabot, full SBOM deferred)
- **Formal compliance certifications** (SOC2, ISO27001) — deferred

## 2. Authentication

### 2.1 User API Keys

**Format:** `oa_live_<32_byte_urlsafe_random>` (prefix `oa_live_` for live, `oa_test_` for test environments — grep-friendly for leak detection)

**Storage:**

```sql
CREATE TABLE api_keys (
    id             UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    key_prefix     VARCHAR(16) NOT NULL,        -- first 12 chars, for display ("oa_live_abcd...")
    key_hash       VARCHAR(256) NOT NULL,       -- argon2 hash of the full key
    user_id        VARCHAR(255) NOT NULL,
    name           VARCHAR(255),                -- human label
    scopes         JSONB NOT NULL DEFAULT '{}', -- { "agents": ["file_agent"], "tools": ["*"], "rate_tier": "standard" }
    created_at     TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    expires_at     TIMESTAMPTZ,
    last_used_at   TIMESTAMPTZ,
    revoked_at     TIMESTAMPTZ,
    rotated_from   UUID REFERENCES api_keys(id)
);

CREATE INDEX idx_api_keys_user ON api_keys(user_id) WHERE revoked_at IS NULL;
CREATE INDEX idx_api_keys_prefix ON api_keys(key_prefix);
```

**Verification:**

1. Extract bearer token from `Authorization: Bearer <key>` header (or `X-API-Key`)
2. Look up candidate rows by `key_prefix` (first 12 chars) — avoids scanning all keys
3. argon2 verify against `key_hash`
4. Reject if `revoked_at` set or `expires_at` in past
5. Cache `(key_prefix, hash_result)` in Redis for 60s to avoid argon2 cost on every request

**Creation / rotation:**

- Admin API endpoint `POST /admin/v1/users/{id}/api-keys` generates new key
- **Plaintext returned once, never stored** — UI displays and warns user to save it
- Key rotation: create new key, 48h overlap, revoke old. Automated via scheduled job (monthly default, configurable per-tenant).
- Leak detection: if a log scanner finds a string starting with `oa_live_` in log output or git history, alert fires and the key is auto-revoked

**Scope semantics:**

```json
{
  "agents": ["file_agent", "search_agent"],    // or ["*"]
  "tools": ["read_file", "search_web"],        // or ["*"] — enforced at MCP Gateway
  "rate_tier": "standard",                      // standard | premium | unlimited
  "expires_at": "2026-10-01T00:00:00Z",
  "ops_admin": false                            // false for user keys; admin keys use separate table
}
```

Scope enforcement happens at multiple layers:
- `agents` enforced by orchestrator: if user calls a message but scope excludes any dispatch target, orchestrator's `dispatch_to_<name>` tool returns `"not permitted for this user"` to the LLM, which the LLM communicates to the user
- `tools` enforced by MCP Gateway (more robust — catches even if orchestrator is bypassed)
- `rate_tier` maps to rate-limit bucket (§3)

### 2.2 Admin API Keys

**Format:** `oa_admin_<32_byte>`, grep-pattern distinct from user keys.

**Storage:** separate `admin_keys` table; never mix with user keys.

```sql
CREATE TABLE admin_keys (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    key_prefix    VARCHAR(16) NOT NULL,
    key_hash      VARCHAR(256) NOT NULL,
    label         VARCHAR(255) NOT NULL,
    scopes        JSONB NOT NULL,            -- { "ops": ["read"] } or { "ops": ["read", "write"] }
    created_at    TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    expires_at    TIMESTAMPTZ,
    last_used_at  TIMESTAMPTZ,
    revoked_at    TIMESTAMPTZ
);
```

**Middleware:**

- `X-Admin-Key` header required on all `/admin/v1/*` routes
- Never falls through to user-auth path (avoids confused-deputy)
- Write-action scopes (`ops:write`) require explicit scope; read endpoints require only `ops:read`

### 2.3 OAuth / OIDC (deferred)

- SSO via enterprise IdP (SAML / OIDC) is a likely v1.3+ requirement
- Not in v1.2. Adding it later fits cleanly: OIDC issues a bearer, middleware resolves to a user record, rest of the pipeline unchanged.

## 3. Rate Limiting

### 3.1 Tiers

- Set per API key via `scopes.rate_tier`
- Tiers configured in code (not DB) so defaults are reviewable

```python
RATE_TIERS = {
  "standard":   { "requests_per_minute": 60,   "tokens_per_hour": 100_000 },
  "premium":    { "requests_per_minute": 300,  "tokens_per_hour": 500_000 },
  "unlimited":  { "requests_per_minute": None, "tokens_per_hour": None    },
}
```

### 3.2 Dimensions

| Limit | Scope | Key |
|---|---|---|
| Per-user requests | RPM | `ratelimit:rpm:user:{user_id}` |
| Per-user tokens | Hourly | `ratelimit:tph:user:{user_id}` |
| Per-IP requests | RPM (defense against unauth DoS) | `ratelimit:rpm:ip:{ip}` |
| Per-endpoint | RPM (e.g., `/sessions` creates) | `ratelimit:rpm:ep:{endpoint}` |
| Global | RPM (hard ceiling) | `ratelimit:rpm:global` |
| Per-tool (at MCP Gateway) | RPM per `(user, tool)` | `ratelimit:rpm:tool:{tool}:{user_id}` |

### 3.3 Algorithm

Sliding-window counter, atomic via Lua:

```lua
-- redis/scripts/sliding_window.lua
-- KEYS[1] = counter key
-- ARGV[1] = limit, ARGV[2] = window_seconds, ARGV[3] = now_ms
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2]) * 1000
local now = tonumber(ARGV[3])
redis.call("ZREMRANGEBYSCORE", key, 0, now - window)
local count = redis.call("ZCARD", key)
if count < limit then
  redis.call("ZADD", key, now, now)
  redis.call("EXPIRE", key, math.ceil(window / 1000))
  return { 1, limit - count - 1 }
else
  return { 0, 0 }
end
```

### 3.4 Behavior

- Over limit → 429 with `Retry-After` header
- Pre-check at orchestrator entry, pre-check at MCP Gateway for tool calls
- Dry-run mode for new tier rollouts: return `X-RateLimit-Would-Reject: true` header but serve, to observe impact before enforcing

## 4. Tool Access Policy

### 4.1 Enforcement

Tool access is **checked twice** — defense in depth:

1. **Agent-local** (in `ManualAgent._execute_tool`): consult `tool_allowlist_v1` guardrail against agent's registry entry. If the LLM tries to call a tool not in the agent's allowlist → blocked before ever hitting the gateway.
2. **MCP Gateway**: consult `user_tool_access_v1` against the API key's `scopes.tools`. Catches cases where agent allowlist is misconfigured or bypassed.

### 4.2 Configuration

- **Agent allowlist** lives in Postgres `agents.tool_allowlist` (JSONB array of tool names). Authoritative for what an agent *can* call.
- **User scope** lives in API key's `scopes.tools`. Authoritative for what the user can transitively cause to be called.
- Effective set = agent_allowlist ∩ user_scope. Blocked if either excludes the tool.

### 4.3 Egress Controls

Per MCP server config:

```yaml
# services/mcp_gateway/mcp_servers/web_search/config.yaml
egress:
  allowed_domains:
    - "*.wikipedia.org"
    - "arxiv.org"
    - "docs.internal"
  denied_domains:
    - "*.googleapis.com"     # block accidental leaks via Google APIs
  max_redirects: 3
  timeout_s: 15
```

Enforced by the MCP server itself (closest to the actual network call). Gateway verifies the config is present before forwarding the tool call; refuses if missing for any network-capable tool.

## 5. Secrets

### 5.1 Runtime posture (Docker Compose era)

- All secrets via environment variables, loaded from `.env` at container start
- `.env` is **gitignored** (verified in CI — repo scanner blocks commits containing files with `.env` suffix that aren't `.env.example`)
- `.env.example` in repo shows variable names with placeholder values (no secrets)
- Container images never contain `.env`, `.env.local`, or any secret
- Docker Compose `env_file: .env` keeps files out of `docker inspect` logs (more than inline `environment:` would)

### 5.2 Secret inventory

| Secret | Consumer | Rotation frequency | Blast radius |
|---|---|---|---|
| `VLLM_API_KEY` (if set on vLLM) | Orchestrator, all agents | Quarterly | All LLM calls fail on leak; bounded cost on leak within network |
| `POSTGRES_PASSWORD` | Orchestrator, MCP Gateway | Quarterly | Full DB access |
| `REDIS_PASSWORD` | All services | Quarterly | Full Redis access |
| `MEILISEARCH_MASTER_KEY` | Orchestrator, MCP Gateway | Quarterly | Full Meilisearch access |
| `JAEGER_AUTH` (if enabled) | All services | Quarterly | Trace read/write |
| Per-MCP-server upstream creds (e.g., external search API keys) | MCP server | Per upstream policy | External service bill |

### 5.3 Log redaction

`structlog` processor pipeline:

1. JSON-serialize the event
2. **Redaction filter** before emit: regex-replace known sensitive patterns
   - `oa_live_[a-zA-Z0-9]+` → `oa_live_***`
   - `oa_admin_[a-zA-Z0-9]+` → `oa_admin_***`
   - Authorization header values → `***`
   - Any key whose name contains `password`, `secret`, `token`, `api_key` → value replaced with `***`
3. Emit to stdout / file sink

Tests assert: if an emitted log event ever contains a string matching `oa_live_[a-zA-Z0-9]{32,}`, the test fails.

### 5.4 k8s migration path (future)

When moving to Kubernetes (deployment sub-spec, deferred):

- Secrets stored as `Secret` resources, encrypted at rest in etcd
- Distribution via [Sealed Secrets](https://github.com/bitnami-labs/sealed-secrets) — secrets encrypted into git, in-cluster controller decrypts
- Alternative: SOPS with age, GitOps-flavored
- RBAC scopes which pod serviceaccounts mount which secrets (no blanket access)
- Secrets mounted as files (preferred) or env vars (if container doesn't support file loading)

This path is documented here so the migration is planned; implementation is deferred.

## 6. Audit Logging

### 6.1 What gets audited

| Event | Where | Retention |
|---|---|---|
| Tool invocations | `audit_tool_calls` table + OTEL span | 180d |
| Admin actions | `audit_admin_actions` table | 365d |
| Guardrail decisions (blocks + flags) | `audit_guardrail_blocks` table + Redis stream | 180d (Postgres) / 7d (Redis) |
| Authentication events (login, key use, failures) | `audit_auth` table | 365d |
| API key lifecycle (create, revoke, rotate) | `audit_api_key_events` table | forever |

### 6.2 Schemas

```sql
CREATE TABLE audit_tool_calls (
    id              UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    occurred_at     TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    trace_id        VARCHAR(64),
    session_id      VARCHAR(255),
    user_id         VARCHAR(255),
    agent_name      VARCHAR(255),
    tool_name       VARCHAR(255) NOT NULL,
    arguments       JSONB,                  -- redacted of known PII patterns
    result_status   VARCHAR(16),            -- ok | error | blocked
    error_message   TEXT,
    duration_ms     INTEGER
);

CREATE INDEX idx_audit_tool_calls_user_time ON audit_tool_calls(user_id, occurred_at DESC);
CREATE INDEX idx_audit_tool_calls_trace ON audit_tool_calls(trace_id);

CREATE TABLE audit_auth (
    id            UUID PRIMARY KEY DEFAULT gen_random_uuid(),
    occurred_at   TIMESTAMPTZ DEFAULT CURRENT_TIMESTAMP,
    event         VARCHAR(32) NOT NULL,        -- auth_success | auth_fail | key_revoked | key_created
    user_id       VARCHAR(255),
    key_prefix    VARCHAR(16),
    ip            INET,
    user_agent    TEXT,
    details       JSONB
);
```

`audit_admin_actions` and `audit_guardrail_blocks` are defined in Dashboard §2.3 and Guardrails §6.2 respectively.

### 6.3 Integrity

- v1.2: standard Postgres table access controls. Audit tables are write-only for application role; read-only for ops role.
- v1.3+: consider append-only + hash-chain for tamper-evidence (deferred).

### 6.4 PII in audit

- Tool arguments may contain PII. Audit writer runs the same PII-redaction pass (`pii_input_v1`) on arguments before INSERT.
- Result content is NOT written to audit — only status + error message. Full content lives in session history (which has shorter retention).

## 7. Input Hygiene

### 7.1 Pydantic everywhere

Every incoming request body is a Pydantic model. Field-level validation (types, lengths, regex constraints). Reject on parse failure with 422.

### 7.2 SQL

- All database access via SQLAlchemy ORM or parameterized queries
- No string formatting in SQL, anywhere. Tests grep for patterns (`f"SELECT...{`, `.format(...SELECT...`) and fail on any match.

### 7.3 Path handling

File-agent and any tool accepting paths must:
- Resolve to absolute paths
- Assert paths are inside a configured allowed root
- Reject symlinks that escape the root (evaluate after resolution)

### 7.4 JSON payload size

- Hard cap on request body: 1 MB (overridable per-endpoint for file uploads, if added)
- Hard cap on SSE event size: 64 KB
- Hard cap on stored session history: 100 turns; older turns pruned (archived to Postgres `messages` if durable history is needed)

## 8. Dependencies

### 8.1 Python

- `uv` manages dependencies; lockfile (`uv.lock`) committed
- CI runs `pip-audit` on every push; fails build on known high/critical CVEs
- Dependabot (or Renovate) opens PRs for security-relevant updates weekly

### 8.2 Docker base images

- All services use `python:3.13-slim-bookworm` base
- Multi-stage build: build stage installs deps, runtime stage copies only what's needed
- No `root` user at runtime (`USER 10001:10001`)
- CI scans images with Trivy; fails on high/critical

### 8.3 System packages

- Installed packages pinned via apt lockfile where possible
- No `apt install` of anything not explicitly needed

## 9. Network Posture

### 9.1 v1.2 assumptions

- Deployed behind a TLS-terminating reverse proxy (nginx / Traefik); services speak HTTP internally
- Corporate network + firewall is the primary boundary
- No public internet exposure expected for v1.2 deployments

### 9.2 TLS

- All external ingress uses TLS 1.3 (reverse proxy config, not service-level)
- Self-signed cert acceptable for on-prem; CA cert distribution is a deployment concern
- Internal service-to-service: plaintext acceptable for v1.2 (docker-compose network is the boundary). v1.3 considers mTLS when services are on k8s and cross-namespace.

### 9.3 CORS

- Orchestrator user API: `Access-Control-Allow-Origin` configurable per deployment (default: deny)
- Admin API: **never** sends CORS headers (no browser access expected)

### 9.4 Outbound

- Agents and MCP servers can only reach hosts matching their `egress.allowed_domains` (§4.3)
- System-level egress also blocked at firewall where possible (defense in depth)

## 10. Incident Response (v1.2 baseline)

### 10.1 Playbook stubs

Written in `ops/runbooks/`:

- `incident-suspected-key-leak.md` — rotate, invalidate, check audit, notify user
- `incident-tool-abuse.md` — identify, suspend user, dump audit for forensics
- `incident-prompt-injection-attack.md` — review audit, tune pattern set, assess impact
- `incident-data-access-anomaly.md` — freeze access, dump audit, investigate

### 10.2 Notification channels

- Paging channel: defined per deployment (ops team configures at install)
- Security-specific channel: distinct from general ops

## 11. Things Explicitly NOT in v1.2

- **Code executor sandbox** — no code-executor agent ships. Deferred to a dedicated sandbox spec.
- **OAuth / OIDC** — bearer API keys only.
- **Multi-tenancy isolation** — single-tenant assumed (umbrella §16.2).
- **End-to-end encryption of session data** — sessions at rest in Redis are plaintext; rely on Redis access controls + network boundary.
- **Formal SBOM generation** — `pip-audit` and lockfile are the v1.2 story; full SBOM (CycloneDX / SPDX) deferred.
- **Penetration testing automation** — plan to run external pentest before non-scaffold production; not in this doc.
- **Formal compliance** — SOC2 / ISO27001 are deployment-specific; this doc establishes the hygiene that makes them achievable.
